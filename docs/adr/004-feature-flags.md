# ADR-004: Feature Flags на уровне Shell

## Статус
Принято

## Контекст
Система обслуживает разные сценарии: отдельный дилер vs сеть дилеров, с Trade-in или без, разные наборы модулей для разных ролей. Нужен механизм переключения без пересборки.

## Решение
Feature flags управляются shell и передаются модулям через `ModuleContext.featureFlags`.

### Архитектура

```
JSON config / API → FeatureFlagManager (shell) → ModuleContext → Module UI
                         ↓
                    onChange listeners → hot-reload без перезагрузки
```

### Текущие флаги

| Флаг | Описание | По умолчанию |
|------|----------|-------------|
| `network_mode` | Данные одного дилера vs агрегация по всей сети | `false` |
| `trade_in_enabled` | Trade-in функционал в каталоге | `false` |
| `devtools_enabled` | Platform DevTools панель | `true` |

### Как модули реагируют

1. При `mount()` — читают `context.featureFlags`
2. При изменении — shell вызывает `onFeatureFlagsChanged(newFlags)` на активном модуле
3. Если модуль не реализует `onFeatureFlagsChanged` — shell делает unmount + remount

## Последствия

### Положительные
- Один конфиг → разные сценарии без пересборки
- Hot-reload: переключил флаг → UI обновился мгновенно
- Shell контролирует видимость модулей (флаг может скрыть модуль из sidebar)

### Отрицательные
- Модули должны проверять флаги (дополнительный код)
- При большом количестве флагов — комбинаторный взрыв сценариев для тестирования
