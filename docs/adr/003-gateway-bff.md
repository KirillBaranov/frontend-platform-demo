# ADR-003: Gateway BFF (Backend-for-Frontend)

## Статус
Принято

## Контекст
Данные приходят из нескольких источников: legacy Delphi система, новые сервисы, возможно внешние API. Два подхода:

1. **Прямые вызовы** — каждый модуль сам обращается к нужным бэкендам
2. **BFF** — единый прокси, который агрегирует данные и нормализует формат

Прямые вызовы создают проблемы: модули знают о внутренней топологии сервисов, CORS для каждого сервиса, дублирование auth логики, невозможность поменять адрес сервиса без изменения фронта.

## Решение
Gateway BFF — один backend endpoint для всех фронтовых модулей.

```
Модули → /api/* → Gateway → [Delphi, Service B, Service C]
```

### Что делает Gateway
- **Агрегация**: дашборд получает данные из Delphi + другого сервиса в одном запросе
- **Нормализация**: Delphi отдаёт `UPPER_CASE` поля → Gateway нормализует в `camelCase`
- **Auth**: один слой авторизации, модули не хранят credentials к бэкендам
- **Миграция**: заменили Delphi адаптер на новый сервис → фронт не трогаем

### Адаптеры
Gateway использует адаптер-паттерн для каждого источника данных:

```
gateway/src/
  ├── delphi-adapter.ts    ← legacy Delphi (сейчас мок, потом IPC/HTTP)
  ├── catalog-adapter.ts   ← новый сервис каталога (future)
  └── routes.ts            ← маршруты, агрегация
```

## Последствия

### Положительные
- Модули не знают о внутренней архитектуре бэкенда
- Безопасность: нет прямого доступа к внутренним сервисам
- Один формат ошибок, один формат ответов
- Миграция Delphi → новый сервис: меняем только адаптер

### Отрицательные
- Дополнительный сервис для деплоя и поддержки
- Увеличение латентности (один дополнительный hop)
- Gateway может стать bottleneck при высокой нагрузке

### Митигация рисков
- Gateway stateless → горизонтальное масштабирование
- Кэширование на уровне Gateway для тяжёлых запросов
- Circuit breaker при недоступности бэкенда
