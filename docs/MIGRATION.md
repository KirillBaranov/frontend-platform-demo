# Стратегия миграции с Delphi

Постепенная миграция: по одному модулю, без big-bang переписывания.

## Подход: Strangler Fig Pattern

```
Этап 1: Delphi работает, веб-платформа рядом
  Delphi ████████████████████████
  Web    ░░░░ (дашборд)

Этап 2: Часть модулей перенесена
  Delphi ████████████████
  Web    ████████ (дашборд, каталог, заказы)

Этап 3: Всё на веб, Delphi только для legacy данных
  Delphi ████ (адаптер)
  Web    ████████████████████████

Этап 4: Delphi выведен из эксплуатации
  Delphi (отключён)
  Web    ████████████████████████████
```

## Порядок миграции (рекомендация)

| Этап | Модуль | Почему сейчас |
|------|--------|--------------|
| 1 | Дашборд | Read-only, низкий риск, быстрый результат |
| 2 | Каталог | Основной инструмент менеджера, высокий ROI |
| 3 | Заказы | Требует write-операций → нужен надёжный Gateway |
| 4 | Финансы | Критичные данные, требует аудита и тестирования |
| N | Остальные | По приоритету бизнеса |

## Как Gateway помогает в миграции

```
До миграции:
  Module → Gateway → Delphi Adapter → Delphi DB

Во время миграции:
  Module → Gateway → [Delphi Adapter (старые данные) + New Service (новые)]

После миграции:
  Module → Gateway → New Service → New DB
```

Модули не знают о миграции. Gateway переключает адаптеры. Можно мигрировать endpoint за endpoint'ом.

## Data Source режимы

| Режим | Когда использовать |
|-------|-------------------|
| `mock` | Разработка, демо, тесты — работает без backend |
| `http` | Локальная разработка с Gateway + Delphi адаптером |
| `grpc` (future) | Продакшен с новыми сервисами |

Переключение: env variable `VITE_DATA_SOURCE=http|mock|grpc`.

## Риски и митигация

| Риск | Митигация |
|------|-----------|
| Несоответствие данных Delphi/Web | Zod валидация на границе Gateway |
| Delphi недоступен | Circuit breaker в Gateway, fallback на кэш |
| Параллельная работа двух систем | Feature flags: постепенное включение модулей |
| Потеря данных при миграции | Dual-write: писать в обе системы, сверять |
